ملخصات حقن (SQL Injection)
ما هو (SQL Injection)

هو ثغرة أمنية تسمح للمهاجم بإدخال أوامر SQL خبيثة داخل مدخلات المستخدم (مثل نموذج تسجيل الدخول)، ليتم تنفيذها مباشرة على قاعدة البيانات. الوصول إلى جميع بيانات الموقع , سرقة كلمات المرور والمعلومات الحساسة , تعديل أو حذف البيانات , تجاوز تسجيل الدخول

كيف يحدث

يحدث عندما يقوم التطبيق بدمج مدخلات المستخدم مباشرة داخل استعلام SQL بدون تحقق أو حماية.

SELECT * FROM users WHERE username = 'admin' AND password = '1234';

إذا أدخل المهاجم:

' OR '1'='1

يصبح الاستعلام: 

SELECT * FROM users WHERE username = '' OR '1'='1';

انواع (sql injection)
حقن SQL داخل النطاق (In-band SQL Injection)

هو نوع من هجمات SQL Injection يتم فيه استخدام نفس القناة لإرسال الهجوم واستلام النتائج (عادة عبر المتصفح). المهاجم يرسل الاستعلام الخبيث ويشاهد النتيجة مباشرة في الصفحة. الأكثر شيوعًا وسهولة في الاستغلال يعطي نتائج مباشرة وسريعة يمكن من استخراج البيانات بسهولة

كيف يعمل

يحدث عندما يقوم التطبيق بعرض نتائج الاستعلام مباشرة للمستخدم بدون فلترة.

SELECT * FROM products WHERE id = 1;

إذا أدخل المهاجم:

1 OR 1=1

يصبح:

SELECT * FROM products WHERE id = 1 OR 1=1;

النتيجة: عرض جميع المنتجات (تجاوز الفلترة)

حقن SQL العمياء (Blind SQL Injection)

هو نوع من هجمات SQL Injection يحدث عندما لا يعرض التطبيق نتائج الاستعلام أو رسائل الخطأ، لكن يمكن للمهاجم استنتاج البيانات بشكل غير مباشر. لا ترى البيانات مباشرة، لكن تستنتجها من سلوك التطبيق. صعب الاكتشاف من قبل المطورين , لا يعتمد على رسائل الخطأ , يسمح باستخراج البيانات خطوة بخطوة , يُستخدم ضد الأنظمة “الآمنة ظاهريًا”

كيف يعمل

يعتمد على طرح أسئلة على قاعدة البيانات والإجابة تكون

True استجابة طبيعية , False اختلاف في الصفحة أو السلوك

الصفحة تعمل بشكل طبيعي

SELECT * FROM users WHERE id = 1 AND 1=1;

الصفحة تختلف أو لا تعرض بيانات من هنا يبدأ المهاجم ببناء استنتاجاته

SELECT * FROM users WHERE id = 1 AND 1=2;
حقن SQL خارج النطاق (Out-of-band SQL Injection)

هو نوع متقدم من هجمات SQL Injection يتم فيه إرسال البيانات المسروقة عبر قناة خارجية بدلًا من عرضها في نفس التطبيق. المهاجم لا يعتمد على استجابة الموقع، بل يجعل قاعدة البيانات تتواصل مع خادم خارجي يملكه.

كيف يعمل

يقوم المهاجم بحقن أمر يجبر قاعدة البيانات على إرسال طلب إلى سيرفر خارجي (DNS / HTTP)، يحتوي على البيانات الحساسة.

User Input → SQL Query → Database → اتصال بخادم المهاجم → تسريب البيانات
أشهر الطرق

DNS Exfiltration إرسال البيانات عبر طلبات DNS يتم تحويل البيانات إلى جزء من اسم الدومين قاعدة البيانات ترسل طلب DNS يحتوي كلمة المرور

password.attacker.com
ثغرات الويب (Web vulnerabilities)
ثغرة (IDor)

IDOR (Insecure Direct Object Reference) هي ثغرة تحدث عندما يسمح التطبيق للمستخدم بالوصول إلى موارد أو بيانات باستخدام مُعرّف (ID) مباشر بدون التحقق من الصلاحيات. تغيير رقم أو معرف في الرابط قد يمنحك وصولًا لبيانات شخص آخر. الوصول إلى بيانات مستخدمين آخرين تعديل أو حذف بيانات بدون إذن انتهاك الخصوصية قد تؤدي إلى اختراق كامل للحسابات

كيف تحدث

عندما يعتمد التطبيق على ID ظاهر (في الرابط أو الطلب) دون التحقق من أن المستخدم يملك حق الوصول.

https://example.com/profile?id=123

إذا غيّر المهاجم الرقم: 123 الي 124 وإذا لم يتم التحقق من الصلاحيات سيتم عرض بيانات مستخدم آخر

https://example.com/profile?id=124
تجاوز المصادقة (Authentication Bypass)

هو ثغرة أمنية تسمح للمهاجم بالدخول إلى النظام بدون التحقق الصحيح من الهوية أو بتخطي خطوات تسجيل الدخول. بمعنى الوصول للحساب أو النظام دون كلمة مرور صحيحة.

كيف يحدث

يحدث بسبب أخطاء في منطق التحقق أو ضعف في آلية تسجيل الدخول. أشهر طرق تجاوز المصادقة SQL Injection

SELECT * FROM users WHERE username = 'admin' AND password = '1234';

إذا أدخل المهاجم:

' OR '1'='1

يصبح الاستعلام: 

SELECT * FROM users WHERE username = '' OR '1'='1';

ثغرة Command Injection

هي ثغرة تسمح للمهاجم بإدخال أوامر نظام (System Commands) داخل مدخلات التطبيق، ليتم تنفيذها على الخادم. بمعنى المستخدم يرسل بيانات → التطبيق ينفذها كأمر في النظام بدون تحقق. تنفيذ أوامر مباشرة على السيرفر الوصول إلى ملفات حساسة , السيطرة الكاملة على النظام , إمكانية رفع ملفات خبيثة أو حذف بيانات

كيف تحدث

عندما يقوم التطبيق بتمرير مدخلات المستخدم إلى أوامر النظام (مثل shell) بدون فلترة أو حماية.

ping 127.0.0.1

إذا أدخل المهاجم:

127.0.0.1 && whoami

يصبح التنفيذ:

ping 127.0.0.1 && whoami

النتيجة تنفيذ أمر إضافي وعرض المستخدم الحالي

مخاطر ملفات Configuration

هي ملفات تحتوي على إعدادات التطبيق أو النظام مثل: بيانات الاتصال بقاعدة البيانات , مفاتيح API , إعدادات السيرفر

كيف تحدث

ترك الملفات متاحة للعامة Public Access , تخزين بيانات حساسة بدون تشفير , رفعها إلى GitHub أو السيرفر بدون حماية , استخدام إعدادات افتراضية غير آمنة

موقع يحتوي على ملف إعدادات .env داخل السيرفر، لكنه متاح للعامة بالخطأ.

https://example.com/.env
ماذا يحتوي الملف
      DB_HOST=localhost
      DB_USER=admin

      API_KEY=sk-xxxxxx
      APP_DEBUG=true
مفهوم مثلث CIA (السرية، السلامة، والتوافر)

مفهوم مثلث CIA (السرية، السلامة، والتوافر)، والذي يعتبر حجر الزاوية في استراتيجيات الدفاع الرقمي. يهدف هذا النموذج إلى تحديد الأهداف الأساسية لحماية أي نظام معلوماتي، حيث تعمل أضلاعه الثلاثة معاً لضمان بيئة تقنية آمنة وموثوقة. إن فهم هذه المبادئ هو الخطوة الأولى لأي متخصص يسعى لتقييم المخاطر وتصميم الحلول الأمنية الفعالة لمواجهة التهديدات المتزايدة.

السرية (Confidentiality)

تعتبر السرية الركن الأول الذي يهدف إلى منع كشف المعلومات الحساسة للأشخاص أو الأنظمة غير المصرح لها بالوصول إليها. يتطلب تحقيق هذا الهدف استخدام تقنيات متقدمة مثل التشفير القوي للبيانات أثناء سكونها أو انتقالها عبر الشبكات لضمان عدم قراءتها حتى في حال اعتراضها. كما تبرز هنا أهمية سياسات التحكم في الوصول (Access Control) التي تمنح الصلاحيات بناءً على مبدأ "الحاجة إلى المعرفة" فقط. وتعتبر التهديدات مثل هجمات التصيد الاحتيالي أو التجسس الصناعي أكبر التحديات التي تواجه هذا الركن في العصر الحالي. لذا، يتم تعزيز السرية من خلال استخدام المصادقة المتعددة العوامل (MFA) وتدريب الموظفين على حماية بياناتهم.

السلامة أو النزاهة (Integrity)

يركز هذا الركن على ضمان بقاء البيانات دقيقة وصحيحة وكاملة طوال دورة حياتها، وحمايتها من أي تعديل غير مصرح به أو عبث غير مقصود. تشمل السلامة التأكد من أن المعلومات لم يتم تغييرها من قبل متسلل أثناء إرسالها، أو حذف أجزاء منها بسبب خطأ تقني في النظام. لتحقيق ذلك، يتم الاعتماد بشكل كبير على دوال الهاش (Hashing) التي تعطي بصمة فريدة لكل ملف، فإذا تغير محتوى الملف ولو بجزء بسيط، تتغير البصمة فوراً. كما تلعب التواقيع الرقمية دوراً حيوياً في إثبات هوية المرسل وضمان عدم تلاعب أي طرف ثالث بالرسالة الأصلية. وتساعد أنظمة مراقبة التغييرات في اكتشاف أي نشاط مشبوه قد يؤثر على موثوقية قواعد البيانات أو ملفات النظام الحساسة.

التوافر أو المتاحية (Availability)

يعني التوافر ضمان وصول المستخدمين المصرح لهم إلى الأنظمة والبيانات والخدمات في أي وقت يحتاجون إليها دون انقطاع. لا تكتمل المنظومة الأمنية إذا كانت البيانات مشفرة وسليمة ولكنها غير متاحة بسبب عطل في الخوادم أو هجوم حجب الخدمة (DDoS). يتطلب هذا الركن بناء بنية تحتية قوية تعتمد على مبدأ التكرار (Redundancy)، بحيث يتوفر جهاز بديل جاهز للعمل فور تعطل الجهاز الأساسي. كما تشمل استراتيجيات التوافر وضع خطط محكمة للتعافي من الكوارث (Disaster Recovery) والاحتفاظ بنسخ احتياطية دورية يتم اختبارها باستمرار. بالإضافة إلى ذلك، تلعب الصيانة الوقائية للأجهزة وتحديث البرمجيات دوراً كبيراً في منع التوقفات المفاجئة التي قد تؤدي لخسائر فادحة للمؤسسات.

التشفير (Encryption)

هو عملية تحويل البيانات من شكلها الطبيعي (Plaintext) إلى شكل غير مفهوم (Ciphertext) باستخدام خوارزميات رياضية، بحيث لا يستطيع قراءتها إلا الشخص الذي يملك المفتاح الصحيح لفك التشفير. تخيل أنك أرسلت رسالة، لكن بدل أن تكون مفهومة، تصبح رموزًا غريبة لا يمكن قراءتها إلا بمفتاح خاص.

التشفير المتماثل (Symmetric Encryption)

يُعد التشفير المتماثل أحد أقدم وأسرع أنواع التشفير، حيث يعتمد على فكرة استخدام مفتاح واحد فقط لعمليتي تشفير البيانات وفك تشفيرها. تخيل أن لديك صندوقاً لا يمكن قفله أو فتحه إلا بنسخة من نفس المفتاح الفيزيائي. إذا أردت إرسال رسالة سرية:

تقوم بقفل الصندوق باستخدام المفتاح.

ترسل الصندوق إلى الطرف الآخر.

يستخدم الطرف الآخر نفس المفتاح لفتح الصندوق وقراءة الرسالة.

التشفير غير المتماثل (Asymmetric Encryption)

يُعرف التشفير غير المتماثل (Asymmetric Encryption) أيضاً بتشفير المفتاح العام (Public Key Cryptography). على عكس التشفير المتماثل، يعتمد هذا النوع على زوج من المفاتيح الرياضية المرتبطة ببعضها البعض، مما يحل مشكلة "توزيع المفاتيح" يعتمد النظام على مفتاحين مختلفين وظيفياً لكنهما مترابطان رياضياً:

المفتاح العام (Public Key): متاح للجميع. يُستخدم فقط لتشفير البيانات. إذا أراد أي شخص إرسال رسالة مشفرة لك، فإنه يستخدم مفتاحك العام.

المفتاح الخاص (Private Key): سري للغاية ولا يعرفه إلا صاحب العلاقة. يُستخدم فقط لفك تشفير البيانات التي شُفرت بالمفتاح العام المرتبط به. ما يتم تشفيره بالمفتاح العام، لا يمكن فتحه إلا بالمفتاح الخاص المقابل له.

التشفير بالتجزئة (Hashing)

يختلف التشفير بالتجزئة (Hashing) جوهرياً عن الأنواع السابقة؛ فهو ليس "تشفيراً" بالمعنى التقليدي لأنه عملية باتجاه واحد (One-Way). أي أنه بمجرد تحويل البيانات إلى "Hash"، لا يمكن رياضياً استعادة البيانات الأصلية منه. تأخذ خوارزمية التجزئة بيانات بأي حجم (كلمة، صورة، أو حتى ملف ضخم) وتحولها إلى سلسلة نصية ذات طول ثابت

الثبات: مهما كان حجم المدخلات، المخرجات دائماً بنفس الطول (مثلاً 64 حرفاً).

الحتمية: نفس المدخلات ستعطي دائماً نفس الـ Hash تماماً.

تأثير الانهيار (Avalanche Effect): أي تغيير بسيط جداً في المدخلات (تغيير حرف واحد مثلاً) سيؤدي إلى إنتاج Hash مختلف كلياً.

لماذا نستخدم الـ Hashing؟

تخزين كلمات المرور: لا تقوم المواقع الآمنة بتخزين كلمة مرورك كما هي. بدلاً من ذلك، تخزن الـ Hash الخاص بها. عندما تسجل دخولك، يحول الموقع ما كتبته إلى Hash ويقارنه بالمخزن لديه.

التأكد من سلامة الملفات (Checksum): عند تحميل نظام تشغيل (مثل Linux ISO)، يوفر الموقع الـ Hash الخاص بالملف. يمكنك فحص الملف بعد تحميله للتأكد من أنه لم يتلف أو يتم التلاعب به.

التوقيع الرقمي: يتم تجزئة المستند أولاً ثم تشفير الـ Hash الناتج بالمفتاح الخاص لضمان عدم تعديل المحتوى.

تكنولوجيا البلوكشين: تعتمد سلاسل الكتل كلياً على الـ Hashing لربط الكتل ببعضها، مما يجعل تغيير أي بيانات قديمة أمراً مستحيلاً دون كسر السلسلة بأكملها.
